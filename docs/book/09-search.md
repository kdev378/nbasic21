# 第 9 章 探す — 探索アルゴリズム

ここからの 3 章は**アルゴリズム** — 問題の「解き方の手順」を学びます。
アルゴリズムは特定の言語のものではありません。ここで学ぶ手順は
Python でも C でも Excel のマクロでも、まったく同じ形で使えます。
言い換えると、**この 3 章があなたの一生モノの財産**になります。

最初のテーマは「探す」。データの山から目当てのものを見つける
方法です。

## 9.1 線形探索 — 端から順に見る

もっとも素直な探し方: **先頭から 1 個ずつ見ていく**。

```basic
' linear.bas — 線形探索
DIM A%(9)
FOR I% = 0 TO 9
  READ A%(I%)
NEXT I%
DATA 31, 8, 57, 24, 90, 46, 13, 72, 65, 3

INPUT "探す数は"; X%
FOUND% = -1                    ' 見つけた場所 (-1 = まだ見つからない)
FOR I% = 0 TO 9
  IF A%(I%) = X% THEN
    FOUND% = I%
    EXIT FOR                   ' 見つかったら即終了 (もう探す必要なし)
  END IF
NEXT I%

IF FOUND% >= 0 THEN
  PRINT X%; "は"; FOUND%; "番にありました"
ELSE
  PRINT X%; "はありません"
END IF
```

ポイントは 2 つ:

- 「見つけた場所」を入れる変数を、**ありえない値 (-1)** で初期化して
  おく。ループ後にこの変数を見れば「見つかったかどうか」も分かる。
- 見つかった瞬間に `EXIT FOR`。残りを見ても無駄だからです。

線形探索は単純ですが、おどろくほど重要です。なぜなら
**どんなデータにも使える**から。並び順がぐちゃぐちゃでも、
文字列でも、条件が複雑でも (「60 点未満で名前が A で始まる最初の
人」)、端から見ていけば必ず見つかります。

## 9.2 探索にかかる時間を考える

データが 10 個なら、最悪 10 回の比較で答えが出ます。
100 万個なら最悪 100 万回。**データの量に比例して時間が伸びる**
わけです。これをコンピュータ科学では **O(n)** (オーダー・エヌ) と
書きます。「n 個のデータに対して、手間が n に比例する」という意味の
記号です。

かっこいい記号ですが、言っていることは素朴です:

| 書き方 | 意味 | 例 |
|---|---|---|
| O(1) | データが増えても手間は一定 | 配列の 3 番を見る |
| O(log n) | データが倍になっても手間は +1 回 | 二分探索 (次!) |
| O(n) | データに比例 | 線形探索、合計、最大値 |
| O(n²) | データの 2 乗に比例 | 素朴なソート (第 10 章) |

「この手順は、データが 10 倍になったら何倍遅くなる?」と
考える癖 — それが計算量の感覚です。プロの世界では、この感覚が
あるかないかで書くプログラムの質が段違いになります。

## 9.3 二分探索 — 「真ん中」を攻める天才的な方法

第 5 章の数当てゲームを覚えていますか? 1〜100 の数を
「大きい/小さい」のヒントで当てるゲーム。実はあれの必勝法が、
コンピュータ科学の宝物 **二分探索** (バイナリサーチ) です。

必勝法: **常に残り範囲の真ん中を言う**。

- 「50」→ もっと大きい → 答えは 51〜100 に絞られた (半分になった!)
- 「75」→ もっと小さい → 51〜74 に絞られた (また半分!)
- …これを続けると、たった 7 回で 100 個から 1 個を特定できます。
  2 を 7 回かけると 128 > 100 だからです。

データが**小さい順に並んでいる**配列なら、同じ作戦で探せます:

```basic
' binary.bas — 二分探索 (配列は昇順に並んでいること!)
DIM A%(9)
FOR I% = 0 TO 9
  READ A%(I%)
NEXT I%
DATA 3, 8, 13, 24, 31, 46, 57, 65, 72, 90

INPUT "探す数は"; X%
LO% = 0                        ' 探す範囲の左端
HI% = 9                        ' 探す範囲の右端
FOUND% = -1
STEPS% = 0
WHILE LO% <= HI%
  MID% = (LO% + HI%) \ 2       ' 真ん中の位置
  STEPS% = STEPS% + 1
  PRINT "  "; STEPS%; "手目:"; A%(MID%); "と比較"
  IF A%(MID%) = X% THEN
    FOUND% = MID%
    EXIT WHILE
  ELSEIF A%(MID%) < X% THEN
    LO% = MID% + 1             ' 真ん中より右にあるはず
  ELSE
    HI% = MID% - 1             ' 真ん中より左にあるはず
  END IF
WEND

IF FOUND% >= 0 THEN
  PRINT X%; "は"; FOUND%; "番にありました ("; STEPS%; "回の比較)"
ELSE
  PRINT X%; "はありません ("; STEPS%; "回の比較)"
END IF
```

```text
探す数は? 65
   1 手目: 31 と比較
   2 手目: 65 と比較
 65 は 7 番にありました ( 2 回の比較)
```

範囲の左端 `LO%` と右端 `HI%` をじわじわ狭めていき、
左右が入れ違ったら (LO% > HI%)「存在しない」と確定します。

## 9.4 二分探索のすごさを数字で

比較回数を並べてみます:

| データ数 | 線形探索 (最悪) | 二分探索 (最悪) |
|---|---|---|
| 100 | 100 回 | 7 回 |
| 1 万 | 1 万回 | 14 回 |
| 100 万 | 100 万回 | 20 回 |
| 10 億 | 10 億回 | **30 回** |

10 億件から 30 回。これが O(log n) の威力です。辞書で単語を
引くとき、あなたも無意識に二分探索をしています (真ん中へんを
開いて、前か後ろか決める)。ただし条件を忘れずに —
**並んでいるデータにしか使えません**。並べる方法は次章で。

## 9.5 番外編: 数当てゲームの逆襲

こんどはコンピュータに当てさせましょう。あなたが思い浮かべた
1〜100 の数を、コンピュータが二分探索で当てにきます:

```basic
' guess-reverse.bas — コンピュータが当てる番
PRINT "1〜100 の数をこころに思い浮かべてください"
LO% = 1
HI% = 100
DO
  MID% = (LO% + HI%) \ 2
  PRINT "それは"; MID%; "ですか? (h=もっと高い / l=もっと低い / y=当たり)"
  INPUT "答え"; A$
  IF A$ = "y" THEN
    PRINT "やった!"
    EXIT DO
  ELSEIF A$ = "h" THEN
    LO% = MID% + 1
  ELSEIF A$ = "l" THEN
    HI% = MID% - 1
  END IF
  IF LO% > HI% THEN
    PRINT "…答え、途中で変えませんでした?"    ' 矛盾を検出!
    EXIT DO
  END IF
LOOP
```

範囲が空になったら「ウソをつかれた」と分かるのも二分探索の
おもしろいところです。

## 他の言語では

線形探索・二分探索のコードは、どの言語でも今書いたのと
一行一行対応する形になります。実務では言語付属の関数
(Python の `in` や `bisect`) を使いますが、**中身はこれ**です。
中身を一度書いた人は、道具の使いどころと限界 (ソート済み必須、
など) を正しく判断できます。
