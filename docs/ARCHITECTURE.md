# NBASIC-21 コンパイラ内部設計

このドキュメントはコンパイラの内部構造 — パイプライン、中間表現 (IR)、
ランタイム ABI、各バックエンドのコード生成戦略 — を説明する。
言語そのものの仕様は [SPEC.md](SPEC.md) を参照。

## 1. パイプライン全体

```
ソース (.bas)
   │ nbasic/lexer.py      字句解析: 文字列 → トークン列
   ▼                       (行指向: NEWLINE / COLON もトークン)
トークン列
   │ nbasic/parser.py     構文解析: 再帰下降 LL(2) → AST
   ▼
AST (nbasic/ast_nodes.py)
   │ nbasic/analyzer.py   意味解析: 記号表・型検査・ラベル解決・
   ▼                       曖昧ノード (配列 vs 関数) の解決・DATA 収集
型注釈付き AST
   │ nbasic/irgen.py      IR 生成: 制御構造の分解・暗黙変換の挿入・
   ▼                       ランタイム呼び出しへの低水準化
IR (nbasic/ir.py)         型付き三番地コード
   │ nbasic/optimizer.py  (-O 時) 定数畳み込み・コピー伝播・
   ▼                       死コード除去・ジャンプ簡約
最適化済み IR
   ├ nbasic/backend_c.py         → 可搬な C ソース
   ├ nbasic/backend_x64.py       → Windows x64 用 NASM (Microsoft ABI)
   └ nbasic/backend_x64_linux.py → Linux x86-64 用 NASM (System V ABI)
```

`nbasic/driver.py` がこの結線と CLI を提供する。各ステージは前段の
出力だけに依存する純関数的な変換であり、途中結果は `--emit-ir` で
観察できる。

## 2. フロントエンドの設計判断

- **行指向の字句解析** — BASIC では改行が文の区切りなので、
  NEWLINE をトークンとして構文解析器に渡す。`DATA` 文の項目は通常の
  字句規則に従わない生テキストのため、字句解析器が `DATA` キーワード
  を認識した時点で行末まで専用モードに切り替える (`_lex_data_items`)。

- **曖昧構文の遅延解決** — `X(3)` は配列参照とも関数呼び出しとも
  解釈できる。構文解析器は判断せず `IndexOrCall` ノードを作り、
  意味解析器が記号表 (配列 → ユーザー FUNCTION → 組込関数の順) を
  引いて `ArrayRef` / `FuncCall` に置き換える。

- **ラベルは物理行頭のみ** — `IDENT :` を常にラベルにすると
  「引数なし SUB 呼び出し + 文区切り」と衝突するため、行頭に限定する
  (SPEC §4.2)。判定はトークン列上で「直前のトークンが NEWLINE か」を
  見るだけで済み、パーサに状態を持たせない。

- **型検査と変換挿入の分離** — 意味解析器は「この式の型は何か」
  (`expr.ty`) と「代入可能か」だけを判定し、実際の変換命令 (itof/ftoi)
  は IR 生成器が挿入する。AST を書き換えるノード種類が減り、
  型規則が analyzer.py の 1 箇所に集まる。

## 3. 中間表現 (IR)

型付き三番地コード。定義は `nbasic/ir.py`、設計方針は同ファイルの
冒頭コメントにもある。

### 3.1 値

| 種類 | 意味 | 不変条件 |
|---|---|---|
| `VReg` | 仮想レジスタ (式の中間結果) | **一度しか代入されない** (SSA 風)。使用は必ず定義より後 |
| `VarSlot` | 名前付き変数 | 何度でも代入可。`is_global` でデータ領域/フレームを区別 |
| `IntConst` / `FloatConst` | 数値即値 | |
| `StrConst` | 文字列プールの添字 | プールは重複排除される |

VReg の単一代入は最適化器の正しさの根拠になっている: 制御フロー
解析なしで定数伝播ができる (定義に到達しない経路では使用にも
到達しない)。一方 VarSlot のコピーは**伝播しない** — 変数は call や
別の mov で書き換わりうるため (optimizer.py 冒頭コメント)。

### 3.2 命令セット

| 命令 | 形式 | 意味 |
|---|---|---|
| `mov` | d ← a | コピー |
| `add sub mul` | d ← a op b | `ty` が INTEGER なら 64bit ラップ、DOUBLE なら IEEE |
| `fdiv` | d ← a / b | 浮動小数除算 |
| `and or xor not` | | 64bit ビット演算 |
| `neg` | d ← -a | `ty` で整数/浮動小数を区別 |
| `itof` / `ftoi` | | INTEGER ↔ DOUBLE (ftoi は最近接偶数丸め) |
| `cmp` | d ← (a op b) ? -1 : 0 | `extra` = EQ/NE/LT/LE/GT/GE、`ty` = 比較する型 |
| `label / jmp / jz / jnz` | | 制御フロー |
| `call` | [d ←] sym(args...) | ランタイム / ユーザー関数呼び出し |
| `ret` | [a] | 復帰 |

意図的に**存在しない**もの: 文字列演算 (すべて `call nb_*`)、
整数除算 (`call nb_idiv` — ゼロ除算検査のため)、べき乗
(`call nb_pow`)、配列アクセス (`call nb_arr_*`)、間接ジャンプ。
この最小性により、バックエンドが実装すべき命令は「ロード/ストア・
整数と倍精度の四則・比較・分岐・呼び出し」だけになる。

### 3.3 IR の観察

```
python3 -m nbasic --emit-ir [-O] program.bas
```

## 4. 低水準化 (irgen.py) の要点

### 4.1 制御構造

FOR は「終了値・STEP を隠しスロットに保存し、STEP の符号を毎回判定」
という古典セマンティクス通りに展開する (irgen.py `_g_ForStmt` の
コメントに展開形がある)。SELECT CASE は条件の連鎖に展開し、対象式は
一度だけ評価して隠しスロットに置く。

### 4.2 評価順序の保証

変数をオペランドに使う前に必ず `mov %t, $slot` で仮想レジスタへ
コピーする。`A + F(A)` の F が A を書き換えても、左辺は呼び出し前の
値になる (SPEC §5.1 の左→右評価)。この mov は大半が最適化器で消える
(消してよい場合のみ)。

### 4.3 GOSUB / RETURN

間接ジャンプ命令を IR に持ち込まないため、次の方式を使う:

1. 各 GOSUB 地点にプログラム全体で一意な整数 ID を割り当てる。
2. `GOSUB L` は `call nb_gosub_push(ID)` → `jmp L`、直後に復帰ラベル。
3. `RETURN` は関数末尾の**ディスパッチャ**へ跳ぶ。ディスパッチャは
   `nb_gosub_pop()` した ID を自関数内の全 GOSUB 地点と比較連鎖で
   照合し、一致した復帰ラベルへ跳ぶ。

ID スタックはランタイムの可変長配列で、GOSUB 再帰も動く。

### 4.4 配列 = ランタイム管理のハンドル

配列変数の実体は「ランタイム内の配列表への 1 起点 ID (int64)」で、
IR/バックエンドはただの整数として扱う。確保 (`nb_arr_new1/2`)、
添字検査、要素の読み書き (`nb_arr_get/set{1,2}_{i64,f64,str}`) は
すべてランタイムが行う。ハンドル 0 は「未確保」を意味し、DIM 前の
アクセスを確実に検出する。

## 5. ランタイム ABI (runtime/nbrt.h)

生成コードとランタイムの契約:

- 値はすべて 8 バイト: `int64_t` / `double` / `nb_str*` (文字列記述子
  へのポインタ)。
- `nb_str` は `{ int64_t len; const char *data; }` の**イミュータブル**
  記述子。文字列リテラルはバックエンドが静的データとして記述子ごと
  埋め込む。
- 実行時に作られる文字列・配列はブロックアリーナから割り付けられ、
  終了時に一括解放される (個別 free なし)。
- 生成コード側が定義する契約シンボル: `nb_basic_main` (メイン関数)、
  `nb_data_items` / `nb_data_count` (DATA 項目表)。`main()` は
  ランタイム側にあり、引数の保存とコンソール初期化 (Windows の VT
  モード有効化) を行ってから `nb_basic_main()` を呼ぶ。
- 真理値は -1/0、DOUBLE→INTEGER は最近接偶数丸め (`llrint` /
  `cvtsd2si` が一致することを利用)。
- **出力チャネル**: PRINT 系関数は「現在の出力チャネル」(0 = 画面、
  1〜15 = OPEN されたファイル) に書く。`PRINT #n` は生成コードが
  `nb_set_channel(n)` で囲むだけで、書式処理・桁ゾーン追跡のコードを
  画面とファイルで共有できる。
- **端末制御**: CLS/LOCATE/COLOR は ANSI エスケープシーケンスで実装し、
  INKEY$ だけが OS 依存 (Windows: `_kbhit`/`_getch`、POSIX: termios の
  非カノニカルモード)。標準入力が端末でないときは「次の 1 バイトを
  読む」動作に落ちるため、パイプ入力でテストできる。

## 6. バックエンド

### 6.1 C バックエンド (backend_c.py)

IR と C はほぼ 1:1 (スロット → ローカル変数、label/jmp → goto、
call → 関数呼び出し)。注意点は 1 つだけで、**符号付きオーバーフローの
未定義動作**を避けるため加減乗算・符号反転を `uint64_t` 経由で行う。
最適化はホストの C コンパイラに任せる — 高速な実行物が欲しい場合は
こちらのバックエンドを使う。

### 6.2 x86-64 バックエンド (backend_x64.py / backend_x64_linux.py)

出力は NASM 構文。ターゲットは 2 つ:

- **Windows x64** (`-t x64`) — Microsoft x64 呼び出し規約
- **Linux x86-64** (`-t x64-linux`) — System V AMD64 ABI

**命令選択・データ生成・フレーム構成は両者で完全に共通**なので、
共通基盤を基底クラス `X64Backend` (backend_x64.py) に置き、ABI 依存
箇所だけを 5 つのフック (ヘッダ・引数領域サイズ・引数の受け取り・
call の引数搬送・ELF/COFF 固有セクション) としてサブクラスが実装する。
Win64 と SysV の本質的な違いは呼び出し規約だけである:

| | Windows x64 | System V (Linux) |
|---|---|---|
| 整数引数 | rcx rdx r8 r9 (4 個) | rdi rsi rdx rcx r8 r9 (6 個) |
| 浮動小数引数 | xmm0-3 (**位置対応**) | xmm0-7 (**整数と独立に数える**) |
| スタック渡し | 第 5 引数以降 [rsp+32]〜 | レジスタからあふれた分 [rsp+0]〜 |
| シャドウ空間 | 32 バイト必須 | 無し |
| 16B 整列 / 戻り値 | 同じ (rsp≡0 at call / rax・xmm0) | 同じ |

レジスタ割り付けを行わない「スタックマシン式」の素朴なコード生成で、
全スロットをフレームに置き、各命令が ロード → 演算 → ストア する。
フレームレイアウト・スクラッチレジスタの取り決めは backend_x64.py
冒頭のコメントに詳しい。

共通部の要点:

- `push rbp` 直後の rsp が 16 バイト整列であることを利用し、フレーム
  サイズを 16 の倍数にすることで call 時の整列条件を常に満たす
  (両 ABI とも同じ規則)。
- 全引数が 8 バイトなのでスタック引数は型を問わずビットパターンとして
  GPR 経由で運べる。
- 比較は `setcc` + `neg` で 0/1 を 0/-1 に変換 (BASIC 真理値)。
- NASM のローカルラベル (`.` 始まり、直前の通常ラベルごとにスコープが
  切れる) を使い、IR のラベル名に `.` を付けるだけで関数間衝突を防ぐ。

Linux 版の固有事項 (backend_x64_linux.py 冒頭コメント):

- 生成コードは RIP 相対参照だけを使うので PIE としてそのままリンク
  できる。
- GNU ld の実行可能スタック警告を避けるため `.note.GNU-stack`
  セクションを出力する。
- 可変引数関数の AL 規約は、呼び先がすべてプロトタイプ付きの
  ランタイム関数なので不要。

### 6.3 なぜ複数のバックエンドか

C バックエンドは可搬性と速度 (ホスト cc の最適化。macOS や ARM も
この経路)、x64 バックエンドは「IR から機械語直前までを自前で見せる」
教育目的と、C コンパイラなしで実行物を作る経路を担う。全バックエンドは
同じ IR・同じランタイムを共有するので、出力の一致テスト (tests/) が
互いの検証になる — 特に x64-linux はホスト上でネイティブ実行して
C 版と突き合わせられるため、最も安価な検証経路でもある。

## 7. 最適化器 (optimizer.py)

関数単位で 4 パスを不動点まで反復する:

1. 定数畳み込み (64bit ラップ・偶数丸め・-1/0 真理値を実行時と一致させる)
2. コピー伝播 (定数と VReg のみ。VarSlot 源は伝播しない — §3.1)
3. 死コード除去 (純粋命令の未使用結果、到達不能コード)
4. ジャンプ簡約 (定数条件分岐、直後ラベルへのジャンプ、無参照ラベル)

`call` は入出力・実行時エラーの可能性があるため常に保存される。

## 8. テスト戦略 (tests/run_tests.py)

1. **ゴールデンテスト** — examples/ の各プログラムを C バックエンドで
   ビルド・実行し、expected/ の期待出力と比較する。`-O` あり/なしの
   両方を走らせ、最適化が観測可能な動作を変えないことを確認する。
2. **クロス検証** — nasm / x86_64-w64-mingw32-gcc / wine が揃っている
   環境では、同じプログラムを x64 バックエンドでもビルドして Windows
   実行物を wine で実行し、C バックエンドと同一の出力になることを
   確認する (改行 CRLF のみ正規化)。ホストが Linux x86-64 なら
   x64-linux バックエンドでもビルドし、ネイティブ実行で検証する。
3. **エラーテスト** — コンパイルエラー・実行時エラーが正しい位置・
   メッセージで報告されることを確認する。

実行方法:

```
python3 tests/run_tests.py            # 利用可能なツールを自動検出
```
